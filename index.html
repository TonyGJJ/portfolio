AI 工作流（从需求到合入）

下面是我日常可重复执行的一套节奏：目标不是「让 AI 写更多」，而是减少返工、可控 diff、可复盘。工具名可换，顺序建议保留。

01开工前：把上下文「喂对」

  • 仓库里维护可读入口：README / AGENTS.md（或等价）——如何安装、如何启动、常用命令、目录职责、禁止改动的路径。
  • 大需求前先限定阅读范围：让模型列出将涉及的文件清单，确认无误再改，避免「全仓库乱扫」。
  • 需要设计或接口事实时，优先 MCP / 链接到单一事实源（Figma 节点、OpenAPI、Confluence/飞书文档），而不是粘贴半截截图。
  • 团队有自研 MCP时，在 README 或运维小抄里固定写清：如何启用、当前暴露了哪些 tool、各 tool 的用途与示例入参（不含密钥）；新人按文档即可接客户端。

02需求进场：先结构化，再写代码

  • 把 PRD 压成固定段落：目标 / 非目标 / 用户路径 / 接口契约 / 边界与异常 / 验收步骤（可点击路径）。
  • 拆成可独立合并的子任务：每个子任务对应明确的文件或模块边界，避免一个 PR 改 30 个无关文件。
  • 先产出「实现计划」：改动点列表 → 风险点 → 回滚方式；复杂逻辑计划通过后再允许动代码。

03编码循环：小步、可 diff、可验证

  • 使用 @文件 / 选中片段限定编辑范围；要求模型给出变更摘要（改了什么、为什么、影响面）。
  • 默认小步提交：一个逻辑点一次 commit，方便 bisect 与 Code Review。
  • 生成后本地跑类型检查 / Lint / 关键单测或冒烟路径；RN/小程序再跑真机或模拟器关键路径。
  • 重构类任务：先补行为不变的测试或录屏路径，再让 AI 协助搬代码，避免 silent break。

04红线：这些默认不交给模型「一次生成完」

  • 安全与钱： 鉴权、支付、金额计算、优惠券叠加、敏感个人信息（PII）。
  • 数据与并发： 事务边界、幂等、分布式锁、复杂 SQL / 迁移脚本。
  • 密钥： 永不写入仓库；环境变量名可讨论，值不进对话、不进日志。

做法是：模型可以写草稿或测试环境脚本，但合入前必须人工对照业务规则逐条过一遍，必要时补单测或双人复核。

05MCP：选型、接入与自研（我有一套 checklist）

  • 先问「这条信息是否经常变」：常变的用 MCP 或文档深链；一次性的只贴最小片段，避免把整库上下文撑爆。
  • 工具分层：只读优先（查设计、查文档、查 schema）；写操作必须可撤销或有二次确认，并保留调用记录便于复盘。
  • 自研 Server（我做过落地）： 先写「工具说明书」再给模型用——每个 tool 的名称、用途、入参示例、失败时返回什么；实现上与客户端联调通 stdio/SSE，线上再配限流与密钥轮转。
  • 自研时额外注意：最小权限（按角色暴露 tool）、可观测（延迟、错误率）、版本化（tool schema 变更可灰度），避免团队里「同一个 MCP 各用各的方言」。

06团队落地：让效率变成集体资产

  • 沉淀Skills / 提示词模板：Code Review、写测试、写发布说明、接口联调 checklist。
  • 统一「AI 辅助」约定：分支命名、commit message、哪些目录允许 Agent 自动改、哪些必须手写 PR 描述。
  • 周会可加 10 分钟：本周省时的用法 / 翻车的案例，避免每人一套野路子。
